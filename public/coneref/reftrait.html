Traits

Traits look so much like structs, no one would blame you for being confused:

trait Eq
  fn `==`(a Self, b Self) Bool

This trait defines no fields and only a single method that overloads the == operator. Unlike a struct, however, no implementation is given for the method. Even stranger, no variable may declare its value as having the Eq type:

imm anEqValue Eq    // *** ERROR ***

Traits are a different kind of type. Whereas struct, enum, etc. enable the creation of concrete types, trait is used to create abstract types. Abstract types play an indirect role as a catalyst for new capabilities. Using an abstract type (like Eq), we improve the reusability features of any concrete type whose structure conforms to Eq. In particular, traits facilitate:

  • Virtual references, enabling access to fields or methods of values whose concrete types we won't know until run-time.
  • Inheritance, for making it easier to create concrete types that share common implementations for fields and methods.
  • Type markers, to distinguish between types that support a certain behavior and those that forbid it.
  • Generic constraints, to help detect whether a type is supported for expansion by a specific generic type or function.

A trait may loosely be compared to a filter that allows us to see into the fields, methods and constrained behavior of other compliant types. By compliant, we simply mean that for each and every field and method defined by the trait, we can find exactly the same field or method defined in the compliant type. The compliant type may implement more fields and methods, but it must implement at least the ones the trait defines.
